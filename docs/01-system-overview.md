# 01. Общее описание системы

---

## 1. Тип системы

**CRM/ERP-гибрид** — модульная платформа, которая объединяет классическую CRM-функциональность (клиенты, лиды, сделки, воронка продаж) с расширенными ERP-возможностями (проекты, финансы, документооборот, контроль персонала).

## 2. Основные модули

Система состоит из **9 функциональных модулей** + ядро:

| Модуль | Код | Назначение | Приоритет MVP |
|--------|-----|-----------|---------------|
| **CRM (ядро)** | `crm` | Клиенты, контакты, лиды, сделки, воронка | ✅ MVP |
| **Коммуникации** | `comms` | Telegram, Email, телефония — омниканальный сбор и история | ✅ MVP |
| **Задачи и контроль** | `tasks` | Постановка задач, дедлайны, контроль исполнения сотрудниками | ✅ MVP |
| **Проекты** | `projects` | Управление проектами после закрытия сделки | 🔶 Post-MVP |
| **Финансы** | `finance` | Счета, оплаты, задолженности, финансовый учёт | ✅ MVP |
| **Документы** | `documents` | КП, договоры, шаблоны, файлы | ✅ MVP |
| **Маркетинг** | `marketing` | Сегментация, источники, кампании | 🔶 Post-MVP |
| **Календарь** | `calendar` | События, встречи, планирование, напоминания | ✅ MVP |
| **Администрирование** | `admin` | Роли, права, настройки, аудит | ✅ MVP |
| **Аналитика** | `analytics` | Дашборды, отчёты, KPI | ✅ MVP (базовая) |

### 3.2. Жизненный цикл данных (Flow)

```
Заявка из канала
    │
    ▼
┌─ Нормализация (единый формат IncomingRequest)
│    - source_type: telegram | email | phone | web_form
│    - contact_data: {phone, email, name, telegram_id, message}
│    - raw_payload: сырые данные из канала
│
├─ Дедупликация клиента
│    - Поиск по телефону/email/telegram_id
│    - Если найден → привязка к существующему
│    - Если не найден → создание нового Customer
│
├─ Создание Лида (Lead)
│    - Статус: "Новый" → Авто-распределение (round-robin или по правилам)
│    - Уведомление менеджеру
│
├─ Квалификация и обработка
│    - Менеджер квалифицирует: спам/брак → закрытие
│    - Валидный лид → конвертация в Сделку (Deal)
│    - Создание связанных задач (follow-up, звонок, КП)
│
├─ Ведение сделки
│    - Перемещение по стадиям воронки
│    - Генерация КП, отправка клиенту
│    - Коммуникации (звонки, письма, сообщения)
│    - Все события фиксируются в Timeline
│
├─ Закрытие сделки
│    - Успешно → выставление счёта → переход в Проект
│    - Отказ → анализ причин (lost reason)
│
├─ Проект (post-sale)
│    - Создание проекта из сделки
│    - Назначение ответственных
│    - Постановка задач, этапов, контроль
│    - Финансовое закрытие (акты, оплаты)
│
└─ Финансовое закрытие
     - Счёт → Оплата → Акт (через 1С-интеграцию)
     - Закрытие проекта
```

### 3.3. Принципы проектирования потоков

1. **Single Source of Truth (SSOT)** — каждая сущность существует в единственном экземпляре; все модули ссылаются на ядро, дублирования данных нет
2. **Event-Driven Processing** — все входящие заявки проходят через Event Bus, что обеспечивает асинхронность и отказоустойчивость
3. **Timeline-First** — любая коммуникация/действие автоматически фиксируется в истории клиента/лида/сделки
4. **Idempotency** — повторная обработка одного и того же входящего сообщения не создаёт дублей (через `external_id` + хэш)
5. **Loose Coupling** — модули связаны через сервисный слой и события, а не через прямые обращения к БД друг друга
