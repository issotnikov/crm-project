# 00. Допущения и ограничения

> Раздел явно перечисляет все допущения, принятые при проектировании системы, так как исходные требования не покрывают всех деталей реализации.

---

## 1. Организационные допущения

| # | Допущение | Обоснование |
|---|-----------|-------------|
| A1 | Размер компании: **10–500 сотрудников**, работающих в CRM одновременно | Средний B2B-бизнес, требующий CRM, но не enterprise-масштаба |
| A2 | Количество активных клиентов: **до 100 000** записей | Достаточно для B2B сегмента |
| A3 | Объём заявок: **до 5 000 в месяц** (пиково до 500/день) | Типичная нагрузка для среднего B2B |
| A4 | Компания оказывает **услуги** (не только продаёт товары), поэтому есть модуль «Проекты» после продажи | Раздел «Проекты» описан как post-sale управление |
| A5 | Мультитенантность **не требуется** на первом этапе (single-tenant). Поддерживается через отдельные инсталляции | MVP ориентирован на одну компанию; архитектура должна позволять добавить multitenancy позже |
| A6 | Основной язык интерфейса — **русский**. Поддержка нескольких языков не требуется в MVP | Локальный рынок |
| A7 | Рабочий день сотрудников: **09:00–18:00 МСК**, пн–пт. Вне рабочее время — автоответчики и отложенные уведомления | Стандартный график |
| A8 | Интеграция с внешними ERP/бухгалтерскими системами (1С) — **через REST API / webhooks**, не через прямое подключение к БД | Изоляция систем |

## 2. Технические допущения

| # | Допущение | Обоснование |
|---|-----------|-------------|
| T1 | Инфраструктура: **собственный сервер / VPS** с Docker. Переход в K8s — на втором этапе | MVP стартует быстро, K8s — для масштабирования |
| T2 | Все компоненты контейнеризованы (Docker) | Портабельность, изоляция |
| T3 | Подразумевается наличие статического публичного IP и доменного имени для webhook'ов (Telegram, телефония, сайт) | Webhook'и требуют публичного endpoint |
| T4 | IP-телефония: **Mango Office, UIScom или Megaplan-телефония** (REST API + webhooks) | Популярные провайдеры на рынке СНГ |
| T5 | Email: сбор через **IMAP** (общие ящики `sales@`, `info@`). Отправка через **SMTP** | Стандартный подход без эксклюзивных API |
| T6 | Telegram: **официальный Bot API** + опционально MTProto (для чтения сообщений из групп) | Bot API покрывает 90% потребностей |
| T7 | Web-форма: интегрируется через **публичный REST API + CSRF-токен** или embed-виджет | Универсальный подход |
| T8 | Браузеры: Chrome, Firefox, Safari, Edge — **последние 2 версии** | Современный веб-стандарт |
| T9 | Бэкап БД: ежедневно (полный), ежечасно (инкрементальный WAL) | Стандартный PITR |

## 3. Функциональные допущения

| # | Допущение | Обоснование |
|---|-----------|-------------|
| F1 | Валюта системы — **рубли (₽)**. Мультивалютность отложена на post-MVP | Один рынок |
| F2 | Налоги: система хранит суммы **без НДС и с НДС**, но не является бухгалтерской системой | Не подменяет 1С |
| F3 | КП и договоры генерируются из **шаблонов** (DOCX/PDF). Сложный редактор документов вне скоупа | Шаблоны покрывают 95% случаев |
| F4 | IP-телефония: запись разговоров хранится во **внешнем хранилище провайдера**, в CRM — только метаданные и ссылка | Экономия места; провайдер хранит аудио |
| F5 | Календарь: **встроенный**, с экспортом в iCal. Прямая синхронизация с Google Calendar — post-MVP | iCal экспорт покрывает базовые потребности |
| F6 | Мобильное приложение — **post-MVP**. Адаптивный веб покрывает мобильные сценарии | PWA на первом этапе |
| F7 | Уведомления сотрудникам: **внутренние (in-app) + email + Telegram** (опционально для каждого сотрудника) | Гибкость каналов оповещения |
| F8 | Маркетинговые рассылки (mass email/telegram) — **отдельный модуль**, не входящий в базовый workflow обработки заявок | Выделен в отдельную сущность |
| F9 | Дедупликация клиентов: автоматическая по **телефону + email**, с возможностью ручного слияния | Предотвращение дублей |
| F10 | Поиск: на MVP — полнотекстовый поиск PostgreSQL (tsvector). Elasticsearch — post-MVP | Упрощение инфраструктуры |

## 4. Границы системы (что НЕ входит)

- ❌ Бухгалтерский учёт (1С-интеграция только для обмена данными)
- ❌ Складской учёт и управление товарами (инвентарь)
- ❌ HR-процессы (приём на работу, расчёт ЗП) — только карточки сотрудников
- ❌ Массовые email/Telegram рассылки (email-маркетинг)
- ❌ Построение сайтов и лендингов
- ❌ Сложная BI-аналитика (OLAP-кубы) — встроенная аналитика на реляционных данных
- ❌ Юридическое согласование документов (эл. подпись) — хранение сканов/файлов

## 5. Допущения по безопасности

| # | Допущение |
|---|-----------|
| S1 | Авторизация — **JWT** (access + refresh токены) |
| S2 | Ролевая модель доступа (**RBAC**) — роли + гранулярные права по модулям |
| S3 | Все действия с чувствительными данными логируются (**audit log**) |
| S4 | Персональные данные хранятся **в зашифрованном виде** на уровне БД (pgcrypto для полей) |
| S5 | API доступно только по **HTTPS** (TLS 1.2+) |
| S6 | Двухфакторная аутентификация (**2FA**) — опционально, рекомендуется для админов |
| S7 | Соответствие **152-ФЗ «О персональных данных»** — физическое хранение данных в РФ |
