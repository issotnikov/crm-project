# 10. Риски и улучшения

## 1. Технические риски

| # | Риск | Вероятность | Влияние | Стратегия митигации |
|---|------|:-----------:|:-------:|---------------------|
| T1 | **Telegram Bot API лимиты** (30 msg/sec, rate limits на getUpdates) | Средняя | Среднее | Long polling с правильным timeout; переход на webhook при росте; очередь исходящих с rate limiter |
| T2 | **Потеря писем при IMAP polling** (письмо удалено до poll) | Низкая | Высокое | IMAP IDLE вместо poll; минимальный интервал 30 сек; сохранение UID; backup ящика |
| T3 | **Дубликаты клиентов** при одновременной обработке | Средняя | Среднее | Distributed lock (Redis) при дедупликации; UNIQUE constraints в БД; транзакционная обработка |
| T4 | **Перегрузка БД** при росте interactions (50K+/мес) | Высокая | Высокое | Партицирование по месяцам; индексы; read-реплики; архивация старых данных |
| T5 | **Генерация PDF медленная** (LibreOffice headless) | Средняя | Среднее | Отдельная Celery очередь; кэширование шаблонов; pre-render; альтернатива: WeasyPrint |
| T6 | **Утечка данных** (персональные данные клиентов) | Низкая | Критическое | Шифрование at-rest (pgcrypto); TLS; RBAC; audit log; 152-ФЗ compliance; регулярные security audits |
| T7 | **Сбой Redis** (потеря очередей задач) | Низкая | Высокое | Redis Persistence (AOF + RDB); Redis Sentinel (HA); AOF rewrite; backup Redis |
| T8 | **Несовместимость API телефонии** (разные провайдеры) | Средняя | Среднее | Adapter pattern; абстракция `TelephonyProvider` интерфейс; поддержка 2–3 провайдеров |
| T9 | **Memory leaks** в Celery workers при долгой работе | Средняя | Среднее | `--max-tasks-per-child=100`; мониторинг memory (Prometheus); перезапуск по расписанию |
| T10 | **Блокировка Telegram-бота** (нарушение правил Telegram) | Низкая | Высокое | Соблюдение ToS; rate limiting; возможность быстрого переключения на нового бота |
| T11 | **Рост времени отклика** при увеличении объёма данных | Высокая | Высокое | Профилирование запросов; EXPLAIN ANALYZE; индексы; read-реплики; кэширование (Redis); Elasticsearch |
| T12 | **Сложность деплоя** без K8s | Средняя | Низкое | Docker Compose для MVP; CI/CD pipeline; Infrastructure as Code (Ansible); переход на K8s позже |

## 2. Бизнес-риски

| # | Риск | Вероятность | Влияние | Стратегия митигации |
|---|------|:-----------:|:-------:|---------------------|
| B1 | **Низкое внедрение** (сотрудники не используют CRM) | Высокая | Высокое | Интуитивный UI; обучение; обязательное использование (управленческое решение); quick wins |
| B2 | **Качество данных** (неполные/некорректные данные) | Высокая | Среднее | Обязательные поля; валидация; авто-дедупликация; регулярный data cleansing |
| B3 | **Гибкость vs Простота** (перегрузка функциями) | Средняя | Среднее | MVP-first подход; принцип 80/20; кастомные поля вместо hardcoded сущностей |
| B4 | **Зависимость от каналов** (Telegram меняет API, email-провайдер отключает) | Средняя | Высокое | Adapter pattern; множественные провайдеры; fallback-каналы; мониторинг доступности |
| B5 | **Сопротивление контролю** (сотрудники против прозрачности) | Средняя | Среднее | Позиционирование как инструмент помощи, а не надзора; gamification (рейтинги, бонусы) |
| B6 | **Превышение бюджета/сроков** | Высокая | Высокое | MVP-скоуп; еженедельные спринты; приоритизация; регулярные ревью прогресса |
| B7 | **Конкурентные решения** (Bitrix24, amoCRM) | Высокая | Среднее | Фокус на уникальных преимуществах: омниканальность с первого дня, гибкость, self-hosted |
| B8 | **Изменение требований** в процессе разработки | Высокая | Среднее | Agile-подход; гибкая архитектура; модульность; backlog management |
| B9 | **Потеря ключевого разработчика** | Средняя | Высокое | Документация; code review; pair programming; распределение знаний в команде |
| B10 | **Соответствие 152-ФЗ** (персональные данные) | Низкая | Критическое | Хранение в РФ; согласие на обработку ПДн; политика конфиденциальности; назначение ответственного |

## 3. Риски масштабирования

| # | Риск | Триггер | Стратегия |
|---|------|---------|-----------|
| S1 | **Деградация производительности БД** | > 100 concurrent users | Read-реплики; pgBouncer; оптимизация запросов |
| S2 | **Очереди переполняются** | > 500 лидов/день | Auto-scaling workers; приоритизация очередей |
| S3 | **Хранилище файлов переполняется** | > 100 GB attachments | Lifecycle policies (S3); архивация; cleanup cron |
| S4 | **WebSocket connections limit** | > 1000 online users | Sticky sessions; горизонтальное масштабирование WS gateway |
| S5 | **Сложность обновлений** (downtime) | > 50 users online 24/7 | Blue-green deployments; rolling updates; backward-compatible migrations |

## 6. План непрерывности бизнеса (BCP)

| Сценарий | Действие | Время восстановления |
|----------|---------|---------------------|
| Сбой сервера приложения | Переключение на backup-сервер / рестарт контейнеров | < 15 мин |
| Сбой PostgreSQL | Восстановление из backup (PITR) | < 1 часа |
| Сбой Redis | Restart; AOF recovery | < 5 мин |
| Сбой хранилища файлов | Восстановление MinIO из backup | < 30 мин |
| Потеря данных | PITR восстановление на точку во времени | < 2 часов |
| Сбой провайдера телефонии | Переключение на резервного провайдера | < 1 часа |
| Блокировка Telegram бота | Создание нового бота, обновление token | < 30 мин |

## 7. Чек-лист перед production-запуском

- [ ] TLS сертификаты установлены и валидны
- [ ] Бэкапы настроены и протестированы (восстановление проверено)
- [ ] Мониторинг работает (Prometheus + Grafana)
- [ ] Алерты настроены (Telegram/email дежурному)
- [ ] Логирование работает (Loki / ELK)
- [ ] Секреты в .env, не в коде
- [ ] RBAC настроен, admin-аккаунты защищены
- [ ] Rate limiting включён
- [ ] Security headers настроены (Nginx)
- [ ] WAF настроен (ModSecurity / Cloudflare)
- [ ] CORS whitelist настроен
- [ ] База данных оптимизирована (индексы, VACUUM)
- [ ] Тестовое покрытие > 70%
- [ ] Документация актуальна
- [ ] План отката (rollback) протестирован
- [ ] Согласие на обработку ПДн (152-ФЗ) подготовлено
- [ ] Политика конфиденциальности опубликована
